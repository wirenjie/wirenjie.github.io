多线程进阶管理

¶线程优先级

  1. Java提供一个线程调度器来监控程序中启动后进入就绪状态的所有线程

  2. 线程调度器根据优先级决定应该优先调度哪个线程来执行

  3. 线程的优先级用数字表示，范围为1~10；

  4. 使用对应方法可以改变和获取优先级

总结：

  1. 线程的优先级低只意味着获得调度的概率低而已
  2. 并不是优先级低就不会被调用或者最后调用，主要看CPU的调度
  3. 线程优先级的设定需要在启动线程前

¶守护线程（daemon）

  1. 线程分为用户线程和守护线程
  2. 虚拟机必须确保用户线程执行完毕
  3. 虚拟机不用等待守护线程执行完毕
  4. 如：后台记录操作日志、监控内存、垃圾回收等待等

¶线程同步机制

  1. 并发：同一个对象被多个线程同时操作

  2. 并发时对对象的修改需要同步

  3. 同步其实就是一种等待机制

  4. 多个线程同时访问对象时进入对象的等待池形成队列

  5. 前面线程执行完后，下一个线程再使用

¶队列 、锁

  1. 线程同步的形成条件为：队列+锁

  2. 队列提供等待条件，锁提供安全条件

  3. 为了保证数据被访问时的正确性，加入了锁机制synchronized

  4. 当一个线程获得对象的排它锁，独占资源，其他线程必须等待

注意：

  1. 一个线程持有锁会导致其他所有需要此锁的线程挂起
  2. 多线程竞争下，加锁和释放锁会导致比较多的调度时延，引起性能问题
  3. 如果一个优先级高的线程等待一个优先级低的线程释放锁，会导致优先级倒置，引起性能问题

¶同步方法、同步块

¶同步方法

  1. private关键字可以保证数据对象只能被方法访问

  2. 因此提出一套类似private机制，即synchronized关键字

  3. 包括synchronized方法和synchronized块

  4. synchronized方法控制对"对象"的访问，每个对象对应一把锁

  5. 每个synchronized方法都必须获得调用该方法的对象的锁才能执行，否则线程会阻塞

  6. 方法一旦执行则独占锁，直到该方法返回才释放锁，后面被阻塞的线程才能获得这个锁，执行线程

不足：

  1. 若将一个大的方法申明为synchronized将会影响效率
  2. 因为方法中需要修改的内容才需要同步，才需要锁
  3. 锁太多了，就会导致资源浪费

¶同步块

  1. Obj可以是任何对象，但是推荐使用共享资源作为监视器
  2. 同步方法无需指定同步监视器，因为方法的监视器就是this，即对象本身，或者class

同步监视器的执行过程：

  1. 第一个线程访问，锁定同步监视器，执行代码
  2. 第二个线程访问，发现同步监视器被锁定，无法访问
  3. 第一个线程访问完毕，解锁同步监视器
  4. 第二个线程访问，发现同步监视器没有锁，然后锁定并访问

¶死锁

  1. 多个线程各自占有一些共享资源，并且互相等待其他线程占有的资源才能运行
  2. 而导致两个以上的线程都在等待对方释放资源，停止执行的情况
  3. 某一个“同步块”同时拥有“两个以上对象的锁”时，就会发生死锁

产生死锁的必要条件：

  1. 互斥条件：一个资源每次只能被一个进程使用
  2. 请求与保持条件：一个进程因请求资源而阻塞时，对已获得的资源保持不释放
  3. 不剥夺条件：进程已获得的资源，在未使用完之前，不能强行剥夺
  4. 循环等待条件：若干进程之间形成一种头尾相接的循环等待资源关系

注意：

  1. 以上四个必要条件，只需要打破其中任意一个或多个条件
  2. 就可以避免死锁的发生

¶Lock锁

  • JDK5.0开始，Java提供更强大的线程同步机制

  • 通过显式定义同步锁对象来实现同步，同步锁用Lock对象充当

  • Java.util.concurrent.locks.Lock接口是控制多个线程对共享资源进行访问的工具

  • 锁提供了对共享资源的独占访问，每次只能有一个线程对Lock对象加锁，线程开始访问共享资源之前应当先获得Lock对象

  • ReentrantLock类（可重入锁）实现了Lock，拥有与synchronized相同的并发性和内存语义

  • 在实现线程安全的控制中，比较常用的是ReentrantLock，可以显式加锁、释放锁

¶synchronized与Lock对比

  1. Lock是显式锁（需要手动开启和关闭锁），synchronized是隐式锁，出了作用域会自动释放锁
  2. Lock只有代码块锁，synchronized有代码块锁和方法锁
  3. Lock锁，JVM将使用较少的时间调度线程，性能更好，扩展性更好（提供更多的子类）
  4. 优先使用顺序：Lock > 同步代码块（已经进入方法体，分配了相应资源）> 同步方法（在方法体之外）
